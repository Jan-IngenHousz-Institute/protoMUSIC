==================================================
INITIAL PROBLEMS THAT WERE IDENTIFIED
==================================================

Before the refactor, the codebase had these issues:

- Mixed ESP-IDF + Arduino “framework soup”
- BME280 path referenced a missing/nonexistent C API
- RTC path relied on Arduino-era transport assumptions
- No shared I2C owner
- Wi-Fi was only partially wired
- SD card logic mounted/unmounted every call and had unsafe global state
- CLI used blocking fgets(stdin)
- Status LED had mixed ownership and lazy init
- Secrets/certs were committed in source
- General patchwork/AI-generated inconsistency

==================================================
REFACTOR PHASES COMPLETED
==================================================

PHASE 1 — Build/framework cleanup
- ESP-IDF made the primary structure
- C code no longer directly includes Arduino/C++ BME headers
- Project builds successfully

Residual risk after Phase 1:
- BME implementation still Arduino-backed
- Arduino still in build
- Temporary compatibility path still present

PHASE 2 — Shared I2C bus
- Added components/hal/i2c_bus
- HAL owns:
  - I2C port
  - pins
  - speed
  - one-time init
  - mutex
- main initializes I2C once
- RTC and BME use the shared bus model

Phase 2 corrections:
- Removed duplicated BME bus-init semantics
- BME validates against HAL bus config instead of initializing it
- Added i2c_bus_get_config()

Residual risk after Phase 2:
- I2C_device still exposes Arduino-era compatibility surface
- tx(..., stop=false) and scan(TwoWire&) are legacy compatibility behavior

PHASE 3 — BME280 shim cleanup
- Public BME API is plain C
- Arduino/Adafruit implementation hidden in private C++ files
- main and CLI no longer see Arduino types
- BME component dependencies tightened with PRIV_REQUIRES

Phase 3 corrections:
- Public dependency leakage removed
- Address validation improved
- Public comment decoupled from HAL naming

Residual risk after Phase 3:
- BME still depends privately on:
  - Arduino
  - Adafruit BME280 code
  - I2C_device compatibility bridge
- Still passes &Wire internally as a private compatibility artifact

PHASE 4 — RTC migration
- RTC no longer depends on:
  - Arduino Wire
  - Arduino TwoWire
  - I2C_device
- Public RTC API remained stable and C-safe
- RTC now uses HAL-owned I2C + i2c_master_write_read_device()
- Repeated-start register reads are correct

Phase 4 corrections:
- Added <stdbool.h> to public RTC header
- Removed dead transitional wrapper header
- RTC internal transport now fully IDF-native

Residual risk after Phase 4:
- RTC_NXP.h is still internal C++ implementation detail living in exported include directory
- Some RTC features beyond current runtime use are build-verified but not hardware-validated

PHASE 5 — Wi-Fi service refactor
- Wi-Fi service now:
  - uses esp_netif_init()
  - creates event loop
  - initializes driver
  - starts station mode
  - connects with configured values
  - tracks connection state
  - handles reconnects
- app_main() now explicitly does:
  - init
  - start
  - connect

Phase 5 corrections:
- Fixed reconfiguration race during disconnect/reconnect
- Added fatal disconnect handling for bad credentials / auth failures
- Removed dead internal state

Residual risk after Phase 5:
- Initial connect still blocks up to ~10 seconds
- Wi-Fi provisioning is still build-time / local-config, not runtime provisioning

PHASE 6 — SD card service refactor
- SD service now:
  - maintains persistent mount state
  - uses one mutex for state and file access
  - centralizes confirmed SDMMC pin map
  - no longer mounts/unmounts every operation
- Public API made more explicit and C-safe

Phase 6 corrections:
- sdcard_unmount() now checks and propagates unmount failure
- sdcard_read_line_at() distinguishes missing file from other errors
- sdcard_file_exists() uses better error signaling

Residual risk after Phase 6:
- sdcard_is_mounted() returns bool, so lock failure and “not mounted” are indistinguishable
- File I/O holds mutex for full operation (safe, but coarse-grained)
- SD service still has no active application consumer

PHASE 7 — CLI + status LED + init-order cleanup
- CLI moved from blocking fgets(stdin) to esp_console REPL
- Status LED now has explicit init and one owner

Phase 7 corrections:

Residual risk after Phase 7:

- Background logs can still interleave with prompt

- The sensor task was later commented out to reduce noise during debug
