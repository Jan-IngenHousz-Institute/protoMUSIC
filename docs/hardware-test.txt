Phase 6 Hardware Test Plan
===========================

Prerequisites
-------------
- Serial monitor running (USB Serial JTAG, 115200 baud)
- AWS IoT Core console open -> MQTT test client subscribed to #
- esp_prov CLI tool or ESP BLE Provisioning Android/iOS app available
- Three certificate files ready from AWS IoT Core thing creation: ca.pem, device.crt, device.key
- mosquitto_pub or a Python script available for publishing to the inbound command topic

Flash commands (PlatformIO — run from project root):
  pio run -t erase       (equivalent to idf.py erase_flash)
  pio run -t upload      (equivalent to idf.py flash)
  pio device monitor     (serial monitor)


------------------------------------------------------------------------
T0 — Factory reset (run before every full test suite)
------------------------------------------------------------------------
  pio run -t erase
  pio run -t upload

Expected: device starts with empty NVS, boots to BLE provisioning mode.
Serial log: "Device not provisioned — starting BLE provisioning"


========================================================================
BLOCK 1: BLE Provisioning + Reboot (6C)
========================================================================

T1.1 — Provisioning without extras (baseline)
----------------------------------------------
Steps:
  1. Boot device (no NVS).
  2. Open ESP BLE Provisioning app, connect to "AMBYTE", PoP "ambyte123".
  3. Send Wi-Fi credentials only. Complete session.

Expected:
  - Log: "Provisioning complete — rebooting to apply"
  - Device reboots and connects to Wi-Fi.
  - Log: "Wi-Fi connected"
  - Log: "MQTT client initialised (TLS=no)"   <-- no certs yet, TLS disabled


T1.2 — Full provisioning: WiFi + device-config + certs in one BLE session
--------------------------------------------------------------------------
Steps:
  1. Erase flash. Boot device.
  2. Open BLE provisioning session ("AMBYTE", PoP "ambyte123").
  3. Send Wi-Fi credentials via standard endpoint.
  4. Send to "dev-cfg" endpoint:
       {"mqtt_uri":"mqtts://YOUR_ENDPOINT.iot.REGION.amazonaws.com:8883",
        "mqtt_client_id":"thing-001",
        "mqtt_topic_root":"ambyte/prod",
        "device_id":"thing-001"}
  5. Send to "cert-prov" endpoint: 4-byte big-endian total_len header + JSON:
       {"ca_cert":"-----BEGIN CERTIFICATE-----\n...\n-----END CERTIFICATE-----\n",
        "dev_cert":"...",
        "dev_key":"..."}
     (Use a Python script to prepend the 4-byte length header.)
  6. End session.

Expected:
  - Log: "Provisioning complete — rebooting to apply"
  - Single reboot.
  - On next boot: "certs initialised (provisioned=yes)"
  - Log: "MQTT client initialised (TLS=yes)"
  - Log: "MQTT connected"
  - AWS IoT console shows client "thing-001" connected.


T1.3 — Timeout with zero writes, no reboot
-------------------------------------------
Steps:
  1. Erase flash. Boot device.
  2. Open BLE session but do NOT send anything.
  3. Wait 120 seconds for timeout.

Expected:
  - Log: "BLE provisioning timed out with no writes — continuing"
  - NO reboot.
  - Execution continues: I2C init, sensor init etc. all logged normally.


T1.4 — Timeout with partial write, reboot required
----------------------------------------------------
Steps:
  1. Erase flash. Boot device.
  2. Open BLE session. Send cert JSON to "cert-prov" only (no Wi-Fi credentials).
  3. Let session time out.

Expected:
  - s_certs_written flag set by handler.
  - Log: "Provisioning complete — rebooting to apply"   <-- reboot despite timeout
  - On next boot: device enters BLE provisioning again (no Wi-Fi stored).
  - Log: "certs initialised (provisioned=yes)"   <-- certs survived in NVS.

BLOCK 2: MQTT Telemetry — no TLS (6A)
========================================================================
Run after T1.1: WiFi connected, no certs, MQTT up without TLS.
Send commands to topic: <topic_root>/<device_id>/cmd
  e.g.  ambyte/thing-001/cmd   (Kconfig defaults)


T2.1 — Raw inbound log command
-------------------------------
Publish to inbound command topic:
  {"cmd":"log","msg":"hello from phase 6"}

Expected:
  - Log: "inbound cmd: logged"


T2.2 — Read env via MQTT command
---------------------------------
  {"cmd":"read_env"}

Expected:
  - Log: "inbound cmd: T=XX.XXC H=XX.X% P=XXXXXXPa"


T2.3 — Store measurement and publish by ID
------------------------------------------
Steps:
  1. Store a measurement (via Lua script or manual CLI).
  2. Publish:
       {"cmd":"publish_unsynced","type":"env"}

Expected:
  - Log: "published measure_id=X msg_id=Y"
  - AWS IoT console receives on topic:
      ambyte/prod/thing-001/env/X
    with payload:
      {"measure_id":1,"sensor_id":1,"measure_type":"env",
       "timestamp":1700000000,
       "values":{"temperature":25.1,"humidity":60.0,"pressure":101325}}
  - Verify "values" is a JSON object keyed by data_type, NOT an array.
  - Verify device identifier is in the topic, not the payload.


T2.4 — INFLIGHT -> PENDING on disconnect
-----------------------------------------
Steps:
  1. Store 3+ measurements.
  2. Send "publish_unsynced" to put one in flight.
  3. Immediately kill WiFi (pull cable or block SSID) before MQTT_EVENT_PUBLISHED.
  4. Restore WiFi.

Expected:
  - Log: "Wi-Fi disconnected — stopping MQTT"
  - device_commands_on_mqtt_disconnect() transitions header INFLIGHT -> PENDING.
  - After WiFi/MQTT reconnect, "publish_unsynced" picks up the same measurement again.
  - Measurement is NOT stuck as INFLIGHT.


T2.5 — INFLIGHT -> PENDING on reboot
--------------------------------------
Steps:
  1. Store a measurement, trigger publish_unsynced.
  2. Hard-reset the device while QoS-1 publish is in flight.
  3. On next boot inspect DB state.

Expected:
  - No rows remain in INFLIGHT state after boot (persistence init resets them to PENDING).
  - "publish_unsynced" on reconnect finds the measurement as PENDING and uploads it.

NOTE: Verify that sqlite_persistence_init() runs:
  UPDATE measurement_headers SET sync_state=PENDING WHERE sync_state=INFLIGHT
on startup. If not, this is an open issue.


T2.6 — mqtt_status command
----------------------------
While connected:
  {"cmd":"mqtt_status"}
Expected: "inbound cmd: MQTT: connected"

Kill WiFi, send again:
Expected: "inbound cmd: MQTT: disconnected"

BLOCK 3: MQTT Inbound Commands — fragmentation and OOM (6B)
========================================================================

T3.1 — Single-chunk command dispatch
--------------------------------------
All commands in T2.1–T2.6 exercise the single-chunk path. Pass if those pass.


T3.2 — Fragmented payload reassembly
--------------------------------------
Steps:
  Publish a command payload > one MQTT packet (e.g. 600+ byte log message):
    {"cmd":"log","msg":"AAAA...(600 chars)...AAAA"}

Expected:
  - mqtt_client reassembles across multiple MQTT_EVENT_DATA events.
  - Log: "inbound cmd: logged" exactly ONCE (not duplicated).
  - No reassembly errors in log.


T3.3 — Oversized payload: no crash
------------------------------------
Steps:
  Publish a payload of ~64 KB to the inbound command topic.

Expected:
  - Log: "OOM: cannot reassemble XXXXX-byte payload"
  - Device keeps running — no panic, no watchdog reset.
  - Subsequent normal commands still dispatch correctly (send T2.1 to confirm).

BLOCK 6: UART Sensor Querying — Phase 7
========================================================================

Prerequisites
-------------
- At least one AMBIT sensor connected to a UART channel (CH0 = AMBIT1
  on GPIO 3 RX / GPIO 46 TX recommended for initial testing).
- Serial monitor running on USB Serial JTAG (not UART — no conflict).
- WiFi provisioned and connected (from Phase 6 tests).

Pin reference:
  CH0 (AMBIT1): UART1  RX=GPIO3   TX=GPIO46  (dedicated)
  CH1 (AMBIT2): UART2  RX=GPIO17  TX=GPIO18  (dedicated)
  CH2 (AMBIT3): UART0  RX=GPIO47  TX=GPIO48  (shared, remapped)
  CH3 (AMBIT4): UART0  RX=GPIO40  TX=GPIO41  (shared, remapped)


T6.0 — Boot auto-ping
----------------------
Steps:
  1. Connect one AMBIT sensor to CH0. Leave CH1-CH3 disconnected.
  2. Flash and boot.

Expected log:
  "UART sensors initialized"
  "  AMBIT1: connected"
  "  AMBIT2: disconnected"
  "  AMBIT3: disconnected"
  "  AMBIT4: disconnected"
  "BOOT: BME280=... UART=OK"

Verify: free heap logged after UART init is reasonable (>100KB).


T6.1 — Ping via CLI
--------------------
  ping_uart 0

Expected: "AMBIT1: connected"

  ping_uart 1

Expected: "AMBIT1: disconnected"  (nothing connected on CH1)

Repeat ping_uart 0 within 10 seconds:
Expected: instant response (cached result, no UART traffic).


T6.2 — UART status
-------------------
  uart_status

Expected: "AMBIT1:connected AMBIT2:disconnected AMBIT3:disconnected AMBIT4:disconnected"


T6.3 — Read temperature
------------------------
  ambit_temp 0

Expected: "AMBIT1 leaf=XX.XC  chip=XX.XC"
  - leaf should be near ambient (15-35C typical).
  - chip should be near or slightly above ambient.

Failure modes:
  - "AMBIT1 get_temp failed" → sensor not responding or wrong baud rate.
  - Timeout → check TX/RX wiring (swapped?), check 115200 baud on both sides.


T6.4 — Read spectrum
---------------------
  ambit_spec 0

Expected: "AMBIT1 spectrum: N N N N N N N N N N  PAR=X.XX"
  - 10 uint16 channel values + PAR float.
  - With sensor in ambient light, values should be nonzero.
  - In dark conditions values should be near zero.


T6.5 — Read sensor info
------------------------
  ambit_info 0 2

Expected: "AMBIT1 info(2): NN bytes" + "FW version: X.X.X"
  - Verify FW version matches what is flashed on the AMBIT sensor.

  ambit_info 0 1

Expected: "AMBIT1 info(1): NN bytes" + "Name: AmbitV00X"
  - Verify the calibration name matches the sensor label.


T6.6 — Blink for identification
--------------------------------
  ambit_blink 0 1 50

Expected:
  - AMBIT sensor's AS7341 LED blinks visually.
  - CLI returns "AMBIT1 cmd 5 OK" after blink completes (~2-3s).
  - No timeout errors.


T6.7 — WiFi stability during UART query
-----------------------------------------
Purpose: verify that UART operations do not starve WiFi/MQTT.

Steps:
  1. Open a second terminal or use `mosquitto_sub` to watch MQTT.
  2. Run a query that takes noticeable time:
       ambit_temp 0
  3. During the query, verify:
     a. WiFi remains connected (check "Wi-Fi disconnected" NOT logged).
     b. MQTT remains connected: send {"cmd":"mqtt_status"} to inbound topic.
        Expected reply: "MQTT: connected".

Expected: no WiFi drops, no MQTT disconnections, UART query completes
normally. FreeRTOS scheduler balances both cores without explicit pinning.


T6.8 — Disconnected sensor graceful failure
---------------------------------------------
Steps:
  1. Physically unplug the AMBIT sensor from CH0.
  2. Run:
       ambit_temp 0

Expected:
  - Timeout after ~5s (wake retries exhausted).
  - "AMBIT1 get_temp failed" or similar error message.
  - No panic, no watchdog reset.
  - ping_uart 0 → "AMBIT1: disconnected"
  - Other CLI commands (status, read_env, i2cscan) still work normally.

  3. Plug sensor back in. Wait 10s (ping cache TTL).
  4. Run:
       ping_uart 0

Expected: "AMBIT1: connected" (sensor recovered).


T6.9 — Shared UART0 remap (requires 2 sensors)
-------------------------------------------------
Prerequisites: sensors on CH2 (GPIO 47/48) AND CH3 (GPIO 40/41).

Steps:
  1. Boot. Verify auto-ping:
       "AMBIT3: connected"
       "AMBIT4: connected"
  2. Read temp from both channels sequentially:
       ambit_temp 2
       ambit_temp 3

Expected:
  - Both return valid temperatures (no cross-talk, no data corruption).
  - Log shows "UART0 remapped to ch2" / "UART0 remapped to ch3" at DEBUG level.
  - No mutex deadlocks (both commands complete within 5s each).

  3. In a Lua script, trigger both in rapid succession:
       local t2 = device.ambit_get_temp(2)
       local t3 = device.ambit_get_temp(3)
       device.log("CH2=" .. t2.leaf .. " CH3=" .. t3.leaf)

Expected: both temperatures returned, no errors. The UART0 mutex serialises
access — second query waits for first to release.


T6.10 — Boot strapping pin (GPIO 46) check
--------------------------------------------
Purpose: verify GPIO 46 (AMBIT1 TX, ESP32-S3 boot strapping pin) does
not interfere with boot.

Steps:
  1. With AMBIT sensor connected on CH0 (TX=GPIO46), power-cycle the ESP32-S3.
  2. Observe boot log — device should enter normal firmware (not download mode).

Expected: normal boot. If boot fails with sensor connected, a series resistor
(~1k) on GPIO 46 is needed to isolate during strapping phase.
